## 一、项目定位

**SSS 是一款基于 ESP32-S3 的智能自行车码表（Smart Cycling Computer）**

- **开发框架**：ESP-IDF v5.3.1
- **主控芯片**：ESP32-S3-WROOM-1-N16R8（16MB Flash + 8MB PSRAM）
- **入口文件**：`main/main.c`（`CMakeLists.txt`）
- **项目说明文档**：`GPIO_USAGE_REPORT.md`

## 二、硬件外设布局（引脚映射）

| 模块 | 总线/协议 | 关键引脚 |
|------|----------|---------|
| **LCD**（ILI9341，LovyanGFX + LVGL） | SPI3 | SCLK=18, MOSI=17, MISO=3, CS=7, DC=16, RST=15, BL=8（PWM） |
| **触摸**（FT6336G） | I2C1 | SCL=9, SDA=10, RST=11 |

### 关键机制

1. **DataCenter + Account（发布-订阅模式）**：在 `components/dataproc/utils/datacenter/`，每个数据源（GPS/IMU/MAG/BLE/RGB/SportStatus/...) 是一个 `Account`，支持 `Subscribe / Commit / Publish / Pull / Notify / Timer`，基于 PingPongBuffer 双缓冲。
2. **PageManager（页面栈管理）**：`components/app/utils/pagemgr/`，支持 Push/Pop、页面动画、生命周期（onViewLoad/onViewWillAppear/.../onViewUnload）。
3. **MVC 页面结构**：每个页面拆分为 `XxxModel / XxxView / Xxx`（控制器）三文件。

## 四、主要任务与时序（`main.c`）

| 任务名 | 周期 | 用途 |
|-------|------|------|
| `LVGL` | 1~500ms 动态 | UI 渲染与输入 |

DataProc 的各个 Account 使用自己的 `lv_timer`（50/100/1000ms）驱动数据更新。

## 五、功能概览

- **Dialplate（主表盘）**：速度、单次距离、时间、海拔（IMU height）、温度、卡路里、心率（BLE HR）、踏频（BLE CSCS）；按键含短按 / 长按（开始/暂停/停止记录）。
- **LiveMap（实时地图）**：基于 SD 卡 `/MAP` 的瓦片图，`MapConv/TileConv/TrackFilter` 自建渲染，支持缩放（slider）、轨迹绘制。
- **Compass（指南针）**：基于磁力计 `atan2(my, mx)` + 软件航向偏置 `CONFIG_COMPASS_HEADING_OFFSET_DEG=116°`；带平滑过渡。
- **RGBControl**：13 种效果（彩虹/闪电/火焰/星空/波浪/烟花/心跳/心率同步/螺旋/色块/极光/纯色/关闭），心率同步模式订阅 `SportStatus` Account。
- **BleDevices**：扫描/连接 HR（0x180D）和 CSCS（0x1816）BLE 设备，CSCS 自动识别 Speed/Cadence/Both 模式。
- **SportStatus（`dp_sport.cpp`）**：速度融合（BLE 优先 / GPS 备用）、Haversine 距离、卡路里、平均速度、异常值过滤。

## 六、潜在问题/优化点（已观察到）

1. **`HAL::RGB_Init` 在两处被初始化**：`HAL_Init_C()` + `main.c` 的 `bsp_rgb_led_init(NULL)`；`hal_rgb.cpp:42` 写死 `gpio_num = 20`，配置来源不统一。
2. **`dp_sport.cpp` 时间单位疑似存在错误**：`s_sport_info.single_time += timeElaps`（tick 数），但后续 `MakeTimeString` 按"秒"使用——累计的是 tick 不是秒，单次时间显示可能偏大。
3. **`Dialplate.cpp` 顶部 Trip 显示**：代码注释"single_distance 单位已经是 km"，但 `LiveMap` 里是 `single_distance / 1000`——两处对单位的理解不一致，其中一处 bug。
4. **`Account.h` 中 `#define nullptr NULL`**：可能与 C++ 关键字冲突，污染全局，需谨慎。
5. **`CompassModel` 对 StatusBar 和 MAG 同步订阅**，但 `onEvent` 只处理 StatusBar，MAG 用 Pull 方式，事件路径不统一。
6. **串口调试任务** `bsp_ble_start_scan` 后直接 `vTaskDelay(11000)` 阻塞等待扫描，期间无法响应其他命令，UX 不佳。
7. **`LiveMap.cpp` `UpdateDelay(0)`** 依赖"tick 溢出差值"技巧，可读性差。
8. **GPS 无定位时默认坐标硬编码** 在 `hal_gps.cpp` 与 `config.h` 两处重复定义（37.47, 116.324444）。
9. **RGBControl** 的 `OnApplyButtonClicked` 每次都 Set 所有参数（mode/brightness/speed/enabled），未校验变化，每次切换均触发 BSP 重启动画。
10. **`dp_rgb.cpp` 的 `DP_RGB_Update` 定时器 50ms 会反向从 HAL 拉状态并回写自己**，与"App 层写入→HAL→DP 层再回写"有潜在竞争（已在代码注释中提到"Don't overwrite user selections"）。

### 为什么"连接成功但没速度数据"？

最可能的几种情况（按概率排序）：

1. **设备用 128-bit 自定义服务**（非 0x1816）：扫描显示成功是因为走了名字匹配，但 `device_type=UNKNOWN`，notify 也不订阅，自然无速度。

## 四、如何根据新日志判断故障类型

烧录后让串口监视器开到 `INFO` 等级（默认即可），连接你的速度传感器后，关注以下关键行：

**A. 扫描阶段**

```
Device scanned #x:
  Name: …
  Adv UUID16 (complete): [0x1816]            ← 标准 CSCS 设备
  Adv UUID128[0]: ……                        ← 如果看到 128-bit，就是非标准传感器
```

- 如果只出现 128-bit UUID 而没有 0x1816 → **设备用私有服务**，需要我们按这个 UUID 做定制解析（这也是速度类传感器最常见的"坑"）。

**B. 服务发现阶段**

```
[x] Service: …
    UUID: 0x1816  (full: 00001816-0000-1000-8000-00805f9b34fb)
    [x.y] Characteristic: CSC Measurement
          UUID: 0x2A5B  (full: 00002a5b-...)
          Properties: 0x10 (NOTIFY )
```

- 如果你看不到 `0x2A5B`，而看到一串别的 `(full: …)` UUID → 设备是私有协议，要依那个 UUID 加解析。
- 如果看到 `0x2A5B` 但没有 NOTIFY 属性 → 设备固件问题，用不了。

**C. 订阅后（关键）**

```
Subscribed to CSC Measurement notifications
  CCCD handle: xx
  CCCD value: 0x0001 (NOTIFY enabled)
```
之后 **拨动车轮或踏频轴**，正常应看到：
```
NOTIFY_RX conn=0 attr_handle=xx indication=0 dev_type=1 len=7
[NOTIFY bytes] len=7 | 01 12 00 00 00 34 12
[CSC raw]      len=7 | 01 12 00 00 00 34 12
CSC Measurement received: flags=0x01 (Speed:Yes Cadence:No), len=7 bytes
```

- **不转车轮时没数据是正常的**，CSCS 设备只有"动"才发数据（看门狗也会提示这点）。
- 如果 **看门狗打印了 `[CSC watchdog]` 警告** → 按警告里 4 条可能性排查。
- 如果看到 `NOTIFY dropped: device_type=UNKNOWN` → 服务发现没识别到，设备用了非标服务。
- 如果看到 notify 但 `flags=0x00` → 设备完全没填速度位，固件 bug 或需要 SC Control Point 配置。
- 如果能看到 `[CSC raw]` 原始字节但最终 `Speed: …` 始终为 0 → 字节序或解析偏移对不上，拿 raw hex 对应官方规范逆向即可。
